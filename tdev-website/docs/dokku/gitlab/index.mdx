# Gitlab

Gitlab auf Dokku deployen.

Referenz
: [docs.gitlab.com/install/docker](https://docs.gitlab.com/install/docker/)


<DynamicInput name="app" default='gitlab' monospace/>
<DynamicInput name="domain" default='gitlab.domain.ch' monospace/>
<DynamicInput name="gitlab_image" default='gitlab/gitlab-ee:18.10.0-ee.0' monospace/>

<TemplateCode>
    ```bash title="root@dokku"
    dokku apps:create {{app}}

    dokku domains:set {{app}} {{domain}}

    dokku ports:add {{app}} http:80:80
    # eventually open ssh port aswell. docker options must be used here.
    # See https://github.com/dokku/dokku/issues/2383#issuecomment-242591167 for more details how to do this.
    # dokku docker-options:add {{app}} deploy -p 2222:22

    # create the storage directory with permissions for root 0:0
    dokku storage:ensure-directory --chown root {{app}}

    mkdir -p /var/lib/dokku/data/storage/{{app}}/config
    chown -R 0:0 /var/lib/dokku/data/storage/{{app}}/config
    dokku storage:mount {{app}} /var/lib/dokku/data/storage/{{app}}/config:/etc/gitlab

    mkdir -p /var/lib/dokku/data/storage/{{app}}/logs
    chown -R 0:0 /var/lib/dokku/data/storage/{{app}}/logs
    dokku storage:mount {{app}} /var/lib/dokku/data/storage/{{app}}/logs:/var/log/gitlab

    mkdir -p /var/lib/dokku/data/storage/{{app}}/data
    chown -R 0:0 /var/lib/dokku/data/storage/{{app}}/data
    dokku storage:mount {{app}} /var/lib/dokku/data/storage/{{app}}/data:/var/opt/gitlab

    # configure environment of {{app}} - important to use http:// and not https protocol, since {{app}} runs behind
    # the nginx reverse proxy!
    dokku config:set {{app}} GITLAB_OMNIBUS_CONFIG="external_url 'https://{{domain}}'; nginx['listen_port'] = 80; nginx['listen_https'] = false;"

    # set resource limits for {{app}}
    dokku resource:limit --cpu 4 --memory 4gb --memory-swap 8gb {{app}}

    # set log level to warn to reduce log size [debug, info, warn, error, fatal, unknown]
    dokku config:set --no-restart {{app}} GITLAB_LOG_LEVEL=warn

    # assign image to {{app}}
    dokku git:from-image {{app}} {{gitlab_image}}

    # letsencrypt: add ssl certificate
    dokku letsencrypt:enable {{app}}
    ```
</TemplateCode>

## Admin login

Um das Admin-Passwort zu setzen, kann nach der ersten Installation der folgende Befehl ausgeführt werden:

<TemplateCode>
    ```bash title="root@dokku"
    dokku enter {{app}}

    gitlab-rake "gitlab:password:reset"
    # username: root
    # password: <your password>
    ```
</TemplateCode>

## Gitlab konfigurieren

Gitlab wird über die `gitlab.rb`-Datei konfiguriert. Weil der Ordner __/var/lib/dokku/data/storage/<Val name="app" />/config__ in den Gitlab-Container gemountet ist, können die Konfigurationen direkt auf dem Host-System vorgenommen werden.

:::info[Hinweis: `config.rb`]
Die `config.rb`-Datei wird von Gitlab automatisch beim ersten Start generiert. Dieser Schritt soll also erst nach der erstmaligen Installatione erfolgen.
:::

<TemplateCode>
    ```bash title="root@dokku"
    nano /var/lib/dokku/data/storage/{{app}}/config/gitlab.rb
    ```
</TemplateCode>

<TemplateCode>
```rb title="/var/lib/dokku/data/storage/{{app}}/config/gitlab.rb"
# Performance optimizations
# @see https://docs.gitlab.com/omnibus/settings/memory_constrained_envs/

# Reduce sidekiq concurrency
sidekiq['concurrency'] = 2

postgresql['shared_buffers'] = "256MB"

# use only a single puma server and no worker
puma['worker_processes'] = 0

# LOG Level
registry['log_level'] = 'warn'
gitlab_shell['log_level'] = 'WARN'

# release memory back to os all 1s
gitlab_rails['env'] = {
  'MALLOC_CONF' => 'dirty_decay_ms:1000,muzzy_decay_ms:1000',
  "GITLAB_LOG_LEVEL" => "WARN"
}

gitaly['configuration'] = {
  logging: {
    level: "warn"
  },
  concurrency: [
    {
        'rpc' => "/gitaly.SmartHTTPService/PostReceivePack",
        'max_per_repo' => 3,
    }, {
        'rpc' => "/gitaly.SSHService/SSHUploadPack",
        'max_per_repo' => 3,
    },
  ]
}

gitaly['env'] = {
  'GITALY_COMMAND_SPAWN_MAX_PARALLEL' => '2',
  'MALLOC_CONF' => 'dirty_decay_ms:1000,muzzy_decay_ms:1000'
}

alertmanager['enable'] = false
gitlab_exporter['enable'] = false
gitlab_kas['enable'] = false
node_exporter['enable'] = false
postgres_exporter['enable'] = false
prometheus_monitoring['enable'] = false
prometheus['enable'] = false
puma['exporter_enabled'] = false
redis_exporter['enable'] = false
sidekiq['metrics_enabled'] = false
```
</TemplateCode>

Danach muss die Konfiguration mit folgendem Befehl neu geladen werden:

<TemplateCode>
    ```bash
    # enter the docker container spawned by dokku
    dokku enter {{app}}

    # and the reconfigure inside the container
    gitlab-ctl reconfigure
    ```
</TemplateCode>

:::info[Ressourcen-Management]
Damit Gitlab nicht zu viel Ressourcen verbraucht, können neben den Docker-Ressourcen-Limits auch einige Konfigurationen vorgenommen werden, um den Speicherverbrauch von Gitlab zu reduzieren. Dies ist ist dann wichtig, wenn Gitlab nur für eine Klasse (EF) verwendet wird und die Geschwindigkeit nicht das zentrale Kriterium darstellt.

https://docs.gitlab.com/omnibus/settings/memory_constrained_envs
:::

## Migrieren

Um Gitlab zu aktualisieren, muss sorgfältig von einer Version auf die nächste migriert werden.

Der Migrationspfad kann dabei von der [Gitlab-Versionstoolbox](https://gitlab-com.gitlab.io/support/toolbox/upgrade-path) abgerufen werden.

<DynamicInput name="app" default='gitlab' />

### Vorbereitung

Die Migration erfolgt auf einer Kopie der aktuellen Gitlab-Instanz (verhindert Datenverlust bei Fehlern).

1. Erstelle ein Backup der aktuellen Gitlab-Instanz
    <TemplateCode>
    ```bash
    docker exec -t {{app}}.web.1 gitlab-backup create
    # erzeugt ein Backup unter /var/lib/dokku/data/storage/{{app}}/data/backups
    ```
    </TemplateCode>
2. Aktuelle Gitlab-Instanz stoppen
    <TemplateCode>
    ```bash
    dokku ps:stop {{app}}
    ```
    </TemplateCode>
3. Daten und Konfiguration in ein neues Verzeichnis kopieren
    <TemplateCode>
    ```bash
    cp -a /var/lib/dokku/data/storage/{{app}}/data /var/lib/dokku/data/storage/{{app}}-migrated/data
    cp -a /var/lib/dokku/data/storage/{{app}}/config /var/lib/dokku/data/storage/{{app}}-migrated/config
    ```
    </TemplateCode>
4. Gitlab-Instanz wieder starten
    <TemplateCode>
    ```bash
    dokku ps:start {{app}}
    ```
    </TemplateCode>

:::info[Ab jetzt auf der `*-migrated`-Instanz arbeiten]
Die folgenden Schritte müssen auf der neuen `*-migrated`-Instanz durchgeführt werden, damit die alte Instanz als Backup erhalten bleibt.
:::

### Migration

Die Migration erfolgt in mehreren Schritten, wobei für jede Gitlab-Version des Migrationpfads folgende drei Schritte durchgeführt werden müssen:


<Steps>
1. Docker-Container für die aktuelle Version erzeugen
    - mounten der lokalen Ordner `data` und `config`
    - starten des Containers, was die notwendige Datenbankmigration automatisch durchführt
    - Überprüfen, ob die neue Version korrekt läuft
2. Backup der neuen Instanz erstellen
3. Docker-Container wieder stoppen und löschen
</Steps>

Um bspw. von Version __15.10.0__ auf __16.3.9__ zu migrieren, müssen folgende Schritte durchgeführt werden:

<Steps>
1. Dockercontainer erzeugen
    - starten
        <TemplateCode>
            ```bash
            docker run --detach \
            --name gitlab-migrate-16-3-9 \
            --volume /var/lib/dokku/data/storage/{{app}}-migrated/config:/etc/gitlab \
            --volume /var/lib/dokku/data/storage/{{app}}-migrated/data:/var/opt/gitlab \
            --shm-size 256m \
            gitlab/gitlab-ee:16.3.9-ee.0
            ```
        </TemplateCode>
    - Überprüfen, ob die neue Version korrekt läuft
        ```bash
        docker exec gitlab-migrate-16-3-9 gitlab-ctl status
        # ausgabe so ähnlich wie:
        # run: alertmanager: (pid 1403) 513s; run: log: (pid 1068) 560s
        # run: gitaly: (pid 1332) 522s; run: log: (pid 560) 688s
        # run: gitlab-exporter: (pid 1375) 515s; run: log: (pid 988) 578s
        # run: gitlab-kas: (pid 1351) 516s; run: log: (pid 735) 676s
        # run: gitlab-workhorse: (pid 1363) 516s; run: log: (pid 907) 591s
        # run: logrotate: (pid 498) 702s; run: log: (pid 510) 699s
        # run: nginx: (pid 947) 587s; run: log: (pid 970) 584s
        # run: postgres-exporter: (pid 1416) 512s; run: log: (pid 1096) 554s
        # run: postgresql: (pid 586) 683s; run: log: (pid 597) 682s
        # run: prometheus: (pid 1384) 514s; run: log: (pid 1043) 566s
        # run: puma: (pid 835) 605s; run: log: (pid 846) 602s
        # run: redis: (pid 515) 695s; run: log: (pid 528) 693s
        # run: redis-exporter: (pid 1377) 515s; run: log: (pid 1017) 572s
        # run: sidekiq: (pid 851) 599s; run: log: (pid 860) 598s
        # run: sshd: (pid 34) 722s; run: log: (pid 33) 722s
        ```
2. Backup der neuen Instanz erstellen
    ```bash    
    docker exec gitlab-migrate-16-3-9 gitlab-ctl stop puma
    docker exec gitlab-migrate-16-3-9 gitlab-ctl stop sidekiq
    docker exec -t gitlab-migrate-16-3-9 gitlab-backup create
    ```
3. Docker-Container wieder stoppen und löschen
    ```bash
    docker stop gitlab-migrate-16-3-9
    docker rm gitlab-migrate-16-3-9
    ```
</Steps>

Um die einzelnen Befehle nicht manuell zu tippen, hier ein Hilfsskript, wobei die Versionen und der App-Name angepasst werden müssen:


<DynamicInput name="app" default='gitlab' />
<DynamicInput name="upgrade-path" default='16.3.9 => 16.7.10 => 17.1.8' />

:::info
Der `upgrade-path` kann direkt von der [Gitlab-Versionstoolbox](https://gitlab-com.gitlab.io/support/toolbox/upgrade-path) kopiert und oben eingefügt werden.
:::

<TemplateCode>
```py live_py slim
versions = [f'gitlab/gitlab-ee:{v}-ee.0' for v in '{{upgrade-path}}'.replace(' ', '').split('=>')]
# pulle alle images vorab
for v in versions:
    print(f'docker pull {v}')
print('')

host_dir = '/var/lib/dokku/data/storage/{{app}}/'

for v in versions:
    nr = v.split(':')[1].split('-')[0].replace('.', '-')
    cname = f'gitlab-migrate-{nr}'
    docker = f'''docker run --detach \\\\
  --name {cname} \\\\
  --volume {host_dir}config:/etc/gitlab \\\\
  --volume {host_dir}data:/var/opt/gitlab \\\\
  --shm-size 256m \\\\
  {v}
'''
    docker_status = f'docker exec {cname} gitlab-ctl status'
    print(f'# [ ] migration {v} erledigt? ------------------ \n')
    print(docker)
    print('')
    print(docker_status)
    print('')
    print(f'docker exec {cname} gitlab-ctl stop puma')
    print(f'docker exec {cname} gitlab-ctl stop sidekiq')
    print(f'docker exec -t {cname} gitlab-backup create')
    print(f'docker stop {cname}')
    print(f'docker rm {cname}')
```
</TemplateCode>

Ist dies alles erledigt, können die Ordner `data` und `config` zur neuen Instanz kopiert werden (mit `scp -a` damit die Berechtigungen erhalten bleiben). Alternativ kann auch das Backup in die neue Instanz kopiert werden und mit `gitlab-backup restore BACKUP=<timestamp_and_version_prefix>` wiederhergestellt werden.
